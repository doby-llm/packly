# Packly GitHub Actions CI/CD Plan

## Current repository state

Packly does not yet contain Android project files, so this task intentionally does not add an active workflow under `.github/workflows/` yet. Adding workflow YAML now would guess at Gradle module names and tasks and would likely create a broken CI run.

Observed missing inputs:

- `settings.gradle` or `settings.gradle.kts`
- root `build.gradle` or `build.gradle.kts`
- `gradlew` / `gradlew.bat`
- Android app module path, for example `app/`
- Android manifest and source/test directories

Once those files exist, CI validation should happen only on GitHub-hosted runners. Do not run Android compilation, lint, unit tests, or APK assembly locally.

## Dependency checklist for GitHub-hosted runners

The future workflow should use these runner dependencies:

- Runner: `ubuntu-latest`
- JDK: Temurin 17, unless the Android Gradle Plugin version selected later requires a newer JDK
- Android SDK: installed by the GitHub runner / Android Gradle Plugin as needed
- Gradle: use the checked-in Gradle Wrapper only (`./gradlew`)
- Gradle dependency cache: `gradle/actions/setup-gradle@v4`
- Java setup/cache: `actions/setup-java@v4` with `cache: gradle`
- Artifact upload: `actions/upload-artifact@v4`
- Checkout: `actions/checkout@v4`

Repository prerequisites before enabling CI:

- Commit a Gradle Wrapper generated by a trusted Android/Gradle setup.
- Decide the app module name (`app` is assumed in examples below).
- Decide minimum supported CI checks based on actual Gradle tasks.
- Keep local developer validation limited to file inspection and Git operations; compilation must happen in GitHub Actions.

## Proposed workflow structure

Create `.github/workflows/android-ci.yml` only after Android project files are present.

Recommended triggers:

- Pull requests to `main`
- Pushes to `main`
- Manual `workflow_dispatch` for ad hoc validation and APK artifact generation

Recommended jobs:

1. `quality`
   - Check out repository.
   - Set up JDK and Gradle cache.
   - Run lightweight Gradle checks that do not require emulator hardware.
   - Candidate commands once tasks exist:
     - `./gradlew --no-daemon lintDebug`
     - `./gradlew --no-daemon testDebugUnitTest`
     - Optional: `./gradlew --no-daemon check` if it is known to be safe and not too broad.

2. `apk`
   - Check out repository.
   - Set up JDK and Gradle cache.
   - Build a debug APK on the GitHub-hosted runner.
   - Upload APK artifacts for retrieval with `gh run download`.
   - Candidate command once the module exists:
     - `./gradlew --no-daemon :app:assembleDebug`

## Draft workflow template for later activation

Do not copy this into `.github/workflows/` until the Android project structure exists and task names are confirmed.

```yaml
name: Android CI

on:
  pull_request:
    branches: [main]
  push:
    branches: [main]
  workflow_dispatch:

permissions:
  contents: read

concurrency:
  group: android-ci-${{ github.workflow }}-${{ github.ref }}
  cancel-in-progress: true

jobs:
  quality:
    name: Lint and unit tests
    runs-on: ubuntu-latest
    timeout-minutes: 30

    steps:
      - name: Check out repository
        uses: actions/checkout@v4

      - name: Set up JDK
        uses: actions/setup-java@v4
        with:
          distribution: temurin
          java-version: '17'
          cache: gradle

      - name: Set up Gradle
        uses: gradle/actions/setup-gradle@v4

      - name: Validate Gradle Wrapper
        uses: gradle/actions/wrapper-validation@v4

      - name: Run Android lint
        run: ./gradlew --no-daemon lintDebug

      - name: Run JVM unit tests
        run: ./gradlew --no-daemon testDebugUnitTest

  apk:
    name: Build debug APK
    runs-on: ubuntu-latest
    timeout-minutes: 30
    needs: quality
    if: github.event_name == 'workflow_dispatch' || github.event_name == 'push'

    steps:
      - name: Check out repository
        uses: actions/checkout@v4

      - name: Set up JDK
        uses: actions/setup-java@v4
        with:
          distribution: temurin
          java-version: '17'
          cache: gradle

      - name: Set up Gradle
        uses: gradle/actions/setup-gradle@v4

      - name: Validate Gradle Wrapper
        uses: gradle/actions/wrapper-validation@v4

      - name: Assemble debug APK
        run: ./gradlew --no-daemon :app:assembleDebug

      - name: Upload debug APK
        uses: actions/upload-artifact@v4
        with:
          name: packly-debug-apk
          path: app/build/outputs/apk/debug/*.apk
          if-no-files-found: error
          retention-days: 14
```

## APK retrieval command

After a workflow run has uploaded the APK artifact, retrieve it from a development machine with:

```bash
gh run list --workflow android-ci.yml --limit 10
gh run download <RUN_ID> --name packly-debug-apk --dir ./artifacts/apk
```

## Recommended local developer tooling

These tools improve velocity without violating the no-local-build rule:

- GitHub CLI (`gh`) for inspecting workflow runs and downloading APK artifacts.
- Android Studio for editing, previews, and project setup. Avoid local builds/tests for this project rule set.
- JDK 17 locally only for IDE indexing if needed; CI remains the source of truth for build validation.
- Gradle Wrapper committed to the repository once Android project generation is done.
- Markdown linting/editor support for docs changes.

## Follow-up acceptance criteria before enabling active CI

Create the active workflow only when all of these are true:

- Android project skeleton exists in this repository.
- Gradle Wrapper is committed and executable.
- App module name is confirmed.
- Actual Gradle task names are confirmed from project files.
- Workflow YAML uses GitHub-hosted runners only.
- CI uploads a debug APK artifact named `packly-debug-apk`.
- README includes or links to `gh run download` instructions for APK retrieval.
